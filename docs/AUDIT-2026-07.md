# Full System Audit ‚Äî July 2026

Scope: all 25 SQL migrations, all server actions (`lib/actions/*`), data layer
(`lib/data/*`), auth plumbing (`proxy.ts`, `lib/supabase/*`), storage policies,
and UI role gating. Verification: `eslint` ‚úÖ clean, `tsc --noEmit` ‚úÖ clean,
`next build` ‚úÖ succeeds. **No automated test suite exists** ‚Äî see ¬ß5.

The codebase is in genuinely good shape for its size: RLS-first multi-tenancy,
atomic fee payments via `record_fee_payment()`, single-use expiring invites,
snapshot columns for financial documents, and honest migration comments. The
findings below are what's left.

---

## 1. Relationship / table errors

### 1.1 HIGH ‚Äî Cross-tenant foreign keys are not enforced
Every child table (`attendance`, `exams`, `fee_payments`, `enrollments`,
`subjects.teacher_id`, `classes.teacher_id`, `departments.head_teacher_id`)
references its parent by plain `id`. RLS guarantees the **row's own**
`school_id` matches the caller, but nothing guarantees the **referenced row**
belongs to the same school. A finance/office user who obtains another school's
student UUID can insert a fee payment or exam pointing at that student.

**Fix:** add `unique (id, school_id)` on `students`, `teachers`, `classes`,
then convert child FKs to composite:
```sql
alter table public.attendance
  add constraint attendance_student_school_fkey
  foreign key (student_id, school_id)
  references public.students (id, school_id) on delete cascade;
```

### 1.2 HIGH ‚Äî The gradebook is disconnected from the `subjects` table
`exams.subject_scores` is a JSONB blob keyed by the **hard-coded**
`GRADEBOOK_SUBJECTS` array in `lib/constants.ts`. Adding a subject on the
Subjects page never appears in exams; renaming one orphans historical scores.
`teachers.subjects` is likewise a free-text `text[]`. Three sources of truth
for the same concept.

**Fix (schema):** an `exam_scores` child table
(`exam_id, subject_id ‚Üí subjects, score numeric check 0‚Äì100`), and a
`teacher_subjects` join table. Migrate the JSONB by name-matching.

### 1.3 MEDIUM ‚Äî No per-subject score constraint; `grade`/`total_score` are client-supplied
DB checks cover `attendance_pct` and `test_score` (0‚Äì100), but individual
`subject_scores` values are unconstrained (a crafted PostgREST request can
store 10‚Åπ), and `total_score` / `grade` are writable columns any office or
teacher account can set to anything. Compute them in a trigger or RPC, or at
least add a JSONB validation check.

### 1.4 MEDIUM ‚Äî `subjects` has no per-school unique name
0019 made `classes`, `departments`, and `academic_years` unique per school but
missed `subjects` ‚Äî duplicate "Maths" rows are silently allowed.
**Fix:** `alter table subjects add constraint subjects_school_name_key unique (school_id, name);`

### 1.5 MEDIUM ‚Äî Invoices have no payment linkage or status
`recordInvoicePayment` inserts a receipt but nothing caps the sum of receipts
at `invoices.total`, and no `status` (unpaid/partial/paid) exists anywhere ‚Äî
overpayment is possible and the UI must re-derive status by summing receipts.
**Fix:** an RPC like `record_fee_payment` (lock ‚Üí sum receipts ‚Üí reject
overpayment ‚Üí insert) plus a computed status view.

### 1.6 MEDIUM ‚Äî `expenses.paid` is mutated in place, with a race
`recordExpensePayment` does read ‚Üí add ‚Üí `Math.min` clamp ‚Üí update in the app:
two concurrent payments lose one update, and there is no payment history for
expenses (unlike fees). **Fix:** an `expense_payments` child table + an atomic
RPC, mirroring the fee design.

### 1.7 LOW ‚Äî Global `seq` counters leak cross-tenant volume
`seq bigint generated always as identity` is global, so School B's invoice
numbers reveal how many documents School A created, and each school's numbering
has gaps. Consider per-school document numbering (a counters table inside the
payment/invoice RPCs).

### 1.8 LOW ‚Äî `invites.student_ids uuid[]` has no referential integrity
Validated only inside `create_invite()`; deleted students linger in arrays and
a stale ID is silently linked on join. Acceptable for now ‚Äî worth a join table
if invites grow.

### 1.9 LOW ‚Äî One school per user
`profiles.school_id` is single-valued (acknowledged "v1"): a parent with
children in two schools, or a teacher working at two, cannot be modeled.

## 2. Security weaknesses

### 2.1 HIGH ‚Äî `activity_log` entries can be forged
The insert policy only checks membership; `actor_id` and `actor_name` are
client-supplied columns. Any member ‚Äî including student/parent accounts ‚Äî can
insert log rows impersonating anyone (`actor_name: 'Head Teacher'`), poisoning
the only audit trail. **Fix:** a `before insert` trigger forcing
`actor_id := auth.uid()` and deriving `actor_name` from the profile; restrict
`kind` to an allowlist.

### 2.2 HIGH ‚Äî Backup restore is not atomic and school-name check is broken
`restoreFromBackup` wipes 12 tables then re-inserts via dozens of REST calls.
Any mid-way failure (network, constraint, RLS) leaves the school half-wiped ‚Äî
the code even acknowledges this. Also both `createBackupSnapshot` and the
restore check hard-code `"Sh.Asharow Primary & Secondary School"`, so in the
multi-tenant world every school's backup carries the wrong name and any
school's file passes any other school's check. **Fix:** move wipe+insert into
a single SECURITY INVOKER RPC (one transaction), stamp the real school id/name
into the snapshot, and verify it on restore.

### 2.3 MEDIUM ‚Äî Children's photos live in a public bucket, forever
`avatars` is public: unguessable UUID URLs, but anyone who ever obtains a URL
can fetch a child's photo with no auth, and photos are never deleted when a
student is removed or a photo replaced (orphaned objects accumulate). For data
on minors, prefer a **private bucket + short-lived signed URLs**, and delete
old objects on replace/remove.

### 2.4 MEDIUM ‚Äî Single-use invites have a small race
`join_school()` checks `used_at is null` and later runs an unconditional
`update invites set used_by ‚Ä¶ where id = v.id`. Two users redeeming the same
code concurrently can both join. **Fix:**
`update invites set used_by = auth.uid(), used_at = now() where id = v.id and used_at is null returning id;`
and raise if no row came back (or `select ‚Ä¶ for update` up front).

### 2.5 MEDIUM ‚Äî No Content-Security-Policy
`next.config.ts` sets a good header baseline (XFO, nosniff, HSTS, referrer,
permissions) but no CSP. Add at least
`default-src 'self'; img-src 'self' https://*.supabase.co data:; connect-src 'self' https://*.supabase.co`
(plus Next's required script/style allowances).

### 2.6 LOW ‚Äî Unvalidated `photo_url` and raw DB errors
`saveStudent`/`saveTeacher` accept any string as `photo_url` (should require
your own storage origin), and most actions return raw `error.message` from
Postgres to the UI, which leaks schema details. `bulkImportStudents` has no
row cap and no date/phone validation ‚Äî one malformed `dob` aborts the whole
insert with a raw error.

### 2.7 LOW ‚Äî Operational gaps
- No MFA offering/enforcement for admin & finance accounts (Supabase supports TOTP).
- Open public signup (accounts just float school-less) ‚Äî fine by design, but
  enable Supabase captcha + rate limits to prevent junk-account floods.
- `platform_delete_school` hard-cascades an entire school with no export step.
- No error monitoring (Sentry) and no rate limiting on server actions beyond
  Supabase defaults.

## 3. Things that are done well (keep them)

- RLS-centric tenancy with `current_school_id()` + GUC ignored for signed-in users.
- Atomic `record_fee_payment` (advisory lock, overpayment guard, receipt in
  the same transaction).
- Personal single-use, expiring, email-lockable invites replacing shareable codes.
- Financial snapshot columns (party details survive record deletion) and
  `on delete restrict` protecting payment history.
- `security_invoker` on `student_fee_balances`; exact numeric money math in DB.
- Enrollment history + academic-year dimension; `updated_at` triggers everywhere.
- Storage MIME/size allowlists enforced server-side; safe `next` redirect handling.

## 4. Missing features (senior product view)

**Academic core**
1. **Timetable / period scheduling** ‚Äî classes have no schedule at all.
2. **Per-subject exams & multiple assessments** (quiz/midterm/final weighting)
   ‚Äî currently one gradebook row per student per term.
3. **Promotion workflow** ‚Äî year switch copies the *same* class into the new
   year; there's no "promote Form 1 ‚Üí Form 2" bulk tool.
4. Homework/assignments, and teacher attendance/leave tracking.

**Finance**
5. **Fee structures**: per-term installments, discounts/scholarships, sibling
   rates, late fees ‚Äî today it's one flat `base_fees` per student per year.
6. **Payroll**: salaries are just an expense category; no per-teacher salary
   records or payslips.
7. Invoice statuses + partial-payment tracking (see ¬ß1.5).

**Communication (highest user-visible value)**
8. Announcements + SMS/WhatsApp notifications: fee reminders, absence alerts
   to `parent_mobile` ‚Äî the data is already captured, nothing uses it.

**Platform**
9. **Pagination** ‚Äî every list view loads entire tables (`students`, `exams`,
   `receipts`‚Ä¶). Fine at 200 students; painful at 2,000. Add `.range()` +
   server-side search.
10. **i18n & RTL** ‚Äî a Mogadishu school with an English-only UI; Somali and
    Arabic (RTL) support would be transformative for office staff.
11. Offline tolerance / PWA for unreliable connectivity.
12. Student documents (birth certificate, transfer letters), health/discipline
    records, ID-card generation (PDF pipeline already exists).
13. De-brand the shell: the sidebar hard-codes "Sh.Asharow" while the platform
    is multi-tenant ‚Äî show the current school's name/logo from `schools`.

## 5. Testing & CI (currently zero)

- No unit, integration, or E2E tests; no CI workflow. Minimum viable safety net:
  - **Vitest** unit tests for `lib/grades.ts`, `lib/utils.ts` (safeNext!),
    `parseItems`, CSV import mapping.
  - **pgTAP or SQL scripts** asserting RLS: each role √ó table matrix
    (esp. cross-tenant reads returning 0 rows, teacher class-scoped writes).
  - **Playwright** smoke: login ‚Üí add student ‚Üí record payment ‚Üí receipt PDF.
  - GitHub Actions: `lint` + `tsc` + `build` + tests on every PR.
